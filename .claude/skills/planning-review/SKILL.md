# 設計書レビュー

設計書の妥当性を複数の視点から検証し、実装前に品質を保証する。

## レビューの目的

- **整合性**: セクション間の矛盾がないか
- **完全性**: 必要な情報が網羅されているか
- **実現可能性**: 技術的に実装可能か
- **品質**: 実装フェーズで迷いが出ない十分な詳細度か

---

## 実行手順

### Step 0: 対象設計書の確認

1. レビュー対象の設計書を特定する
   - 引数で指定された場合: そのパスを使用
   - 指定がない場合: 最新の設計書を `~/.claude/designs/{project-name}/` から検索（プロジェクト名はGitリポジトリ名またはカレントディレクトリ名）
2. 設計書を読み込む
3. 設計レベル（L1-L4）を判定する

---

### Step 1: Subagent による自動検証（必須）

Task ツールで検証用の general-purpose Subagent を起動し、以下の観点でチェックする：

#### チェック項目

**1. 整合性チェック**
- [ ] アーキテクチャ概要の「現状」が既存ファイル（architecture.md等）と一致するか
- [ ] ファイル変更一覧がアーキテクチャ変更を反映しているか
- [ ] リスクセクションが技術的課題を網羅しているか
- [ ] 移行計画の依存関係に矛盾がないか
- [ ] 代替案セクションで選択した案が他セクションと整合するか

**2. 完全性チェック**
- [ ] 設計レベルに応じた必須セクションが全て記載されているか
  - L2: 必須8セクション
  - L3/L4: 必須12セクション
- [ ] 移行計画のステップに抜け漏れがないか
- [ ] テスト計画が実装内容を網羅しているか
- [ ] 完了条件（DoD）が明確に定義されているか
- [ ] ディレクトリ構造とファイル変更一覧が対応しているか

**3. 実現可能性チェック**
- [ ] 前提条件（技術的前提、環境前提）が満たされているか
- [ ] 技術的に実現可能か（外部依存、ライブラリの制約、バージョン互換性）
- [ ] 既存パターンとの整合性は取れているか
- [ ] スコープが現実的か（大きすぎないか）

**4. 品質チェック**
- [ ] 問題点が定量的に記述されているか（行数、複雑度など）
- [ ] 図が適切に使用されているか（アーキテクチャ概要など）
- [ ] 設計方針が具体的か（抽象的な表現になっていないか）
- [ ] リスクに対する軽減策が記載されているか

#### Subagent への指示例

```markdown
以下の設計書の妥当性を検証してください：
{設計書のパス}

**検証観点**:
1. 整合性チェック（上記4項目）
2. 完全性チェック（上記5項目）
3. 実現可能性チェック（上記4項目）
4. 品質チェック（上記4項目）

**出力形式**:
- 問題がある場合: 重大度（Critical/Major/Minor）、具体的な指摘、修正案
- 問題がない場合: 各チェック項目の確認結果
```

---

### Step 2: 人間によるレビュー（推奨）

Subagent の検証と並行して、人間が以下の観点でレビューする：

#### レビュー観点

**ビジネス観点**:
- [ ] ビジネス要件との整合性
- [ ] 優先順位は適切か
- [ ] スコープは適切か（やりすぎ/足りない）

**技術観点**:
- [ ] 設計の方向性が適切か
- [ ] 技術選定が妥当か
- [ ] 見落としている考慮事項はないか

**プロジェクト観点**:
- [ ] 実装にかかる工数は現実的か
- [ ] リスクは適切に評価されているか
- [ ] テスト計画は十分か

#### ユーザーへの案内

> Subagent による自動検証を実行しています。
>
> 並行して、以下の設計書を人間の視点でレビューしてください：
> {設計書のパス}
>
> **レビュー観点**:
> - ビジネス要件との整合性
> - 設計の方向性
> - 見落としている考慮事項
>
> レビュー結果をお知らせください。

---

### Step 3: Codex によるレビュー（オプション）

外部の視点（OpenAI モデル）からの評価を得たい場合、`/codex` スキルを使用する。

#### ユーザーへの確認

> Codex によるレビューを実施しますか？（任意）
>
> Codex MCP は外部モデル（OpenAI）の視点から設計書を評価し、以下を提供します：
> - アーキテクチャの妥当性
> - 潜在的なリスク
> - ベストプラクティスとの比較
>
> 実施する場合は「はい」、スキップする場合は「いいえ」と回答してください。

#### 実行方法

ユーザーが「はい」と回答した場合、`Skill` ツールで `codex` を呼び出す。
引数には設計書の分析に必要な情報を渡す：

```
Skill を呼び出し:
  skill: codex
  args: |
    以下の設計書の妥当性を評価してください。
    対象: {設計書のパス}
    評価観点: アーキテクチャの妥当性、潜在的リスク、完全性、ベストプラクティスとの比較、セキュリティ・パフォーマンス
```

---

### Step 4: フィードバックの統合

1. Subagent の検証結果を確認
2. 人間のレビュー結果を確認
3. （オプション）Codex のレビュー結果を確認
4. 指摘された問題点を整理し、優先度付け

#### 問題の分類

| 重大度 | 説明 | 対応 |
|--------|------|------|
| Critical | 設計の根本的な問題、実装不可能 | **必須修正** |
| Major | 重要な欠落、リスクの高い問題 | **修正推奨** |
| Minor | 改善提案、より良い方法の提案 | 任意 |

---

### Step 5: 修正と再検証

#### Critical/Major の問題がある場合

1. 設計書を修正する
2. 修正箇所を明示する
3. `/planning-review` を再実行して再検証

#### Minor の問題のみの場合

1. 必要に応じて設計書を修正
2. 修正しない場合は理由を記録（ADR推奨）

---

### Step 6: レビュー完了の判定

以下の条件を満たしたらレビュー完了：

- [ ] Subagent の検証で Critical/Major の問題が指摘されていない
- [ ] 人間のレビューで承認されている
- [ ] （Codex レビューを実施した場合）重大な懸念が示されていない
- [ ] 指摘された Critical/Major の問題が全て解決されている

#### 完了時の案内

> **設計書レビューが完了しました。**
>
> **検証結果**:
> - Subagent: 問題なし / {問題数}件の問題を修正
> - 人間レビュー: 承認
> - Codex: {実施した/スキップした}
>
> **次のステップ**:
> `/clear` を実行してから `/implement` を呼び出して実装を開始してください。
>
> **設計レベルに応じた実装方法**:
> - L1/L2: そのまま実装
> - L3/L4: implement が自動的にタスク分解を行います

---

## レビュー結果の記録（オプション）

レビュー結果をトレーサビリティのために記録する場合：

```markdown
# {設計書名} レビュー結果

## レビュー実施日
{日付}

## 検証結果

### Subagent 検証
- 整合性: ✅ / ⚠️ {問題内容}
- 完全性: ✅ / ⚠️ {問題内容}
- 実現可能性: ✅ / ⚠️ {問題内容}
- 品質: ✅ / ⚠️ {問題内容}

### 人間レビュー
- レビュアー: {名前}
- 承認: ✅ / ⚠️ {コメント}

### Codex レビュー
- 実施: はい / いいえ
- 主な指摘: {要約}

## 修正内容
{修正した問題と対応内容}

## 承認
レビュー承認者: {名前}
承認日: {日付}
```
