# 実装フェーズ開始

設計ドキュメントとタスクリストに基づいて実装を開始する。

## 実行手順

### Step 1: Worktree確認・作成

> **前提条件**: `git-wt` がインストールされていること
> ```bash
> brew install k1LoW/tap/git-wt
> git config --global wt.basedir ".worktrees"
> ```

1. 現在の作業ディレクトリを確認
2. **既にfeatureブランチのworktreeにいる場合**（`/planning`から継続）:
   - そのまま実装を開始
3. **mainブランチまたは別の場所にいる場合**:
   - `git wt feature/{feature-name}` でworktreeを作成・移動
   - シェル統合がない場合は `cd .worktrees/feature/{feature-name}` で移動

---

### Step 1.5: タスクリストの読み込み

設計ドキュメントとタスクリストを読み込む。

#### 1.5-1: タスクリストファイルの確認

以下の優先順位でタスクリストを探す：

1. `~/.claude/designs/{project-name}/tasks/NNNN_{対象機能}-tasks.md`（`/plan-tasks` で生成）
2. ユーザーが指定したタスクリストファイル

> **プロジェクト名の決定**: Gitリポジトリ名（`basename $(git rev-parse --show-toplevel)`）またはカレントディレクトリ名を使用する。

**タスクリストファイルが存在する場合:**
- ファイルを読み込み、未完了のタスクを特定
- Step 2 へ進む

**タスクリストファイルが存在しない場合:**
- ユーザーに確認：
  > タスクリストファイルが見つかりません。
  >
  > **選択肢:**
  > 1. `/plan-tasks` でタスクリストを生成する（推奨）
  > 2. 設計書から直接実装する（従来方式）

#### 1.5-2: 設計書の読み込み

以下の優先順位で設計ドキュメントを読み込む：

1. `~/.claude/designs/{project-name}/NNNN_{対象機能}.md`
2. `~/.claude/designs/{project-name}/` 内の関連設計書
3. `docs/adr/` 配下の関連ADR

設計ドキュメントから全体像と設計意図を理解する。

---

### Step 2: 実装対象の特定

#### タスクリストがある場合

タスクリストから未完了の Phase を特定する：

```markdown
## Phase 1: {Phase名}  ← 全タスクが [x] なら完了
### Tasks
- [x] 完了済みタスク
- [ ] 未完了タスク  ← これを実装

## Phase 2: {Phase名}  ← Phase 1 完了後に着手
```

**実装対象の決定:**
1. 最初の未完了タスクがある Phase を特定
2. その Phase の Targets/References を確認
3. Tasks を順番に実装

#### タスクリストがない場合（従来方式）

設計書の「移行計画」セクションから実装対象を特定する。

---

### Step 3: 実装計画の確認

ユーザーに実装計画を提示し、確認を取る：

> **実装対象:** Phase {N}: {Phase名}
>
> **Targets（変更対象）:**
> - {ファイル1}
> - {ファイル2}
>
> **References（参照）:**
> - {ファイル1}
>
> **Tasks:**
> - [ ] {タスク1}
> - [ ] {タスク2}
>
> この Phase を実装してよろしいですか？

---

### Step 4: TDDで実装

タスクリストの Tasks を順番に実装する。

```
1. テストを書く（Red）
2. テストを通す最小限の実装（Green）
3. リファクタリング（Refactor）
```

**各タスク完了時:**
- タスクリストファイルのチェックボックスを更新

```markdown
- [x] 完了したタスク  ← [ ] から [x] に変更
- [ ] 次のタスク
```

**コミットメッセージ**: 日本語で記述

---

### Step 5: 設計からの逸脱対応

実装中に設計からの逸脱が必要になった場合：

1. **軽微な逸脱**（実装詳細レベル）
   - 設計ドキュメントを更新
   - コミットメッセージに理由を記載

2. **重大な逸脱**（アーキテクチャ・設計判断の変更）
   - `/adr` スキルでADRを作成
   - 設計ドキュメントを更新
   - 必要に応じてユーザーに相談

---

### Step 6: Phase完了の確認

Phase の全タスクが完了したら以下を確認：

1. 全テストがパスすること
2. Linter/Formatterのエラーがないこと
3. 設計ドキュメントと実装の整合性

```bash
# テスト実行（プロジェクトに応じて）
npm test / pytest / go test ./...

# Lint/Format
npm run lint / ruff check / golangci-lint run
```

---

### Step 7: コード整形（コミット前）

`code-simplifier`エージェントを実行してコードを整形する：

```
Task tool:
  subagent_type: "code-simplifier:code-simplifier"
  description: "Simplify and refine code"
  prompt: "まだコミットされていないコード（unstaged/staged changes）を整形してください"
```

**実行タイミング**:
- テストがすべてパスした後
- コミットする直前

**注意事項**:
- code-simplifierによる変更後、再度テストを実行してグリーンを確認
- 大きな変更が提案された場合は、ユーザーに確認を取る

---

### Step 8: Phase単位でコミット

Phase の全タスクが完了したらコミットする。

```bash
git add .
git commit -m "Phase N: {Phase名}"
```

**コミット粒度:**
- 1 Phase = 1 コミット が基本
- Phase が大きすぎる場合は、論理的な単位で分割してもよい

**タスクリストの更新確認:**
- 全タスクが `[x]` になっていることを確認

---

### Step 9: 次のPhaseまたは完了

#### 次のPhaseがある場合

> Phase {N} が完了しました。
>
> **次のPhase:** Phase {N+1}: {Phase名}
>
> 続けて実装しますか？

Step 2 に戻り、次の Phase を実装する。

#### 全Phaseが完了した場合

ユーザーに以下を報告：
- 実装した内容のサマリー
- 作成/変更したファイル一覧
- 設計からの逸脱があれば、その内容と理由

PRを作成：
```bash
git push -u origin <branch-name>
gh pr create --title "..." --body "..."
```

## TaskCreate/Update の補助的使用

タスクリストファイル（Markdown）を主とするが、セッション内の進捗管理には TaskCreate/Update を併用してもよい。

**使用する場面:**
- 長時間のセッションで進捗を可視化したい場合
- 複数の Phase を並行して進める場合

**注意:**
- `/clear` すると TaskCreate で作成したタスクは消える
- 永続化が必要な場合は Markdown ファイルを使用する

---

## 注意事項

- 設計ドキュメントを「正」として実装する
- タスクリストのチェックボックスを都度更新する
- 実装中に気づいた改善点は、現タスク完了後に別タスクとして提案
- 過度な最適化・抽象化を避け、設計通りのシンプルな実装を心がける
- テストは設計ドキュメントに記載された振る舞いを検証する
