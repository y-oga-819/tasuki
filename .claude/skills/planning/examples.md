## 1. 設計方針

### 良い例

```markdown
## 設計方針

**ViewEvent + ユースケース層によるシンプルなアーキテクチャ**

- Bubble Tea の `tea.Msg` を ViewEvent として活用
- ユースケース層でビジネスロジックを整理
- ドメインモデルはシンプルに保つ
```

→ 一言で方針を示し、3点で具体化

```markdown
## 設計方針

良い設計を目指します。保守性と拡張性を考慮し、適切なアーキテクチャを採用します。
```

→ 具体性がなく、何も決まっていないのと同じ

```markdown
└───────┼─────────────┼───────────────────┘
        ↓             ↓
┌───────────────────────────────────────────┐
│              UseCase Layer                │
```

→ ASCII図またはMermaidで依存の方向を明示

### 良い例（L3/L4: アーキテクチャ変更がある場合）

```markdown
## アーキテクチャ概要

### 現状のアーキテクチャ

現在は3層アーキテクチャ：
- Presentation（CLI/TUI）
- Service（薄いラッパー）
- Domain（エンティティのみ）

問題点：
- Serviceがリポジトリのラッパーに過ぎず、ビジネスロジックの置き場所がない
- Presentationに手続的ロジックが散在（`user_prompt_submit.go`: 140行）

### 提案するアーキテクチャ

4層アーキテクチャに変更：

┌─────────────────────────────────────────┐
│         Presentation Layer              │
│  ┌──────────┐  ┌──────────┐            │
│  │ CLI Hook │  │   TUI    │  ← 薄く    │
│  └────┬─────┘  └────┬─────┘            │
└───────┼─────────────┼──────────────────┘
        ↓             ↓
┌─────────────────────────────────────────┐
│          UseCase Layer (NEW)            │
│  - ビジネスロジックのオーケストレーション  │
│  - トランザクション境界                   │
└───────────────────┬─────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│           Domain Layer                  │
│  - エンティティ + ドメインロジック         │
└───────────────────┬─────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│        Infrastructure Layer             │
│  - リポジトリ実装                        │
└─────────────────────────────────────────┘

### 差分と変更理由

| 変更箇所 | 現状 | 提案 | 理由 |
|---------|------|------|------|
| レイヤー数 | 3層 | 4層（UseCase追加） | ビジネスロジックの明確な置き場所 |
| Service層 | 薄いラッパー | **削除**（UseCaseに統合） | 責務の重複を解消 |
| Presentation | 140行の手続的ロジック | 30行のUseCase呼び出し | 単一責任原則 |
| Domain | データ構造のみ | データ + ドメインロジック | ドメイン駆動設計 |
```

→ 現状・提案・差分の3部構成で記述
→ 設計書だけで全体像が把握でき、妥当性を検証できる

### 悪い例

```markdown
## アーキテクチャ

UIとビジネスロジックとデータ層で構成されています。
```

→ 図がなく関係性が不明、具体的なコンポーネント名もない
→ L3/L4 で現状との差分が不明

## 4. 既存パターンとの整合性

### 良い例

```markdown
## 既存パターンとの整合性

### エラーハンドリング

| 観点 | 内容 |
|------|------|
| 既存パターン | センチネルエラー (`var ErrNotFound = errors.New(...)`) |
| あるべき姿 | 構造体エラーで詳細情報を保持 |
| 採用する設計 | 既存のセンチネルエラーを踏襲 |
| 理由 | 変更箇所が広範囲になり、今回のスコープを超える |

### 命名規則

既存コードでは `Todo` を使用しているが、今回 `Task` にリネームする。
これは既存パターンからの逸脱だが、以下の理由で許容する：
- 影響範囲が限定的（リポジトリ層のみ）
- 用語の一貫性が向上する
```

→ 理想と現実のギャップを明示

### 悪い例

```markdown
## 既存パターンとの整合性

既存のコードに合わせて実装します。
```

→ 何に合わせるのか、理想との差分が不明

## 5. 代替案とトレードオフ

### 良い例

```markdown
## 代替案とトレードオフ

### イベント伝播の方式

| 案 | 概要 | メリット | デメリット |
|----|------|---------|-----------|
| A. コールバック | 直接関数呼び出し | シンプル、デバッグ容易 | 依存が密結合 |
| B. チャネル | Goのchannelで伝播 | 疎結合、非同期対応 | 複雑度増加 |
| C. tea.Msg活用 | Bubble Tea標準の仕組み | フレームワーク一貫性 | TUI以外で使いにくい |

**選定: C. tea.Msg活用**

理由：
- Bubble Teaアプリとして自然な設計
- CLI Hookでは戻り値を無視すれば済む
- 新しい仕組みを導入するコストを避けられる
```

→ 比較表 + 選定理由

### 悪い例

```markdown
## 代替案

いくつかの案を検討した結果、最適な方法を採用しました。
```

→ 何を検討し、何を基準に選んだか不明

## 6. リスク・注意点

### 良い例

```markdown
## リスク・注意点

### 技術的リスク

| リスク | 影響 | 軽減策 |
|--------|------|--------|
| SDK接続のタイムアウト処理 | セッション管理の不整合 | 明示的なタイムアウト設定とリトライ機構 |
| 並行アクセス時の状態競合 | データ不整合 | ミューテックスによる排他制御 |

### 実装時の注意点

- `SessionID` は非同期で確定するため、確定前に参照しないこと
- Hook処理は同期的に実行されるため、重い処理を含めないこと
- ファイル監視のデバウンス間隔は100ms以上を推奨
```

→ リスクと軽減策をセットで記載

### 悪い例

```markdown
## リスク

特になし。
```

→ 本当にリスクがないのか、検討していないのか不明

## 12. テスト・動作確認方法

```markdown
## テスト・動作確認方法

### 自動テスト

| テストファイル | テスト対象 |
|--------------|-----------|
| `usecase/task/create_task_test.go` | タスク作成ユースケース |
| `usecase/task/change_status_test.go` | ステータス変更ユースケース |
| `domain/task_test.go` | Task状態遷移 |

### 手動確認手順

#### シナリオ1: タスク作成フロー
1. TUIを起動する: `tasuki`
2. `n` キーで新規タスク作成
3. タイトルを入力して Enter
4. Backlog カラムにタスクが表示されることを確認

#### シナリオ2: Hook連携
1. Claude Code を起動: `claude`
2. 任意のプロンプトを入力
3. TUIでタスクが自動作成されることを確認

### 確認すべきポイント

- [ ] 既存のタスクデータが破損していないこと
- [ ] SDK経由とHook経由の両方でタスク作成できること
- [ ] エラー時に適切なメッセージが表示されること

### 完了判断基準

- 全自動テストがパス (`go test ./...`)
- 上記手動シナリオが全て成功
- 既存機能のリグレッションがないこと
```

→ 自動テスト + 手動シナリオ + 完了基準
