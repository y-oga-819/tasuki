# 設計書セクション詳細

各セクションで記述すべき内容の詳細ガイド。

---

## レベル別の適用方針

### L1: Patch
設計書作成は不要。変更内容を説明して実装に移行。

### L2: Feature（簡易版）

**必須セクション（8個、連番で記述）**:
1. 設計方針
2. 現状の問題点（簡略化可）
3. アーキテクチャ概要
4. 既存パターンとの整合性
5. ディレクトリ構造
6. ファイル変更一覧
7. 移行計画
8. テスト・動作確認方法

**状況に応じて追加（9以降の番号で追加）**:
- 選択肢がある → 代替案とトレードオフ
- 技術的難易度が高い → リスク・注意点
- スコープが曖昧 → スコープ外の明示
- 外部依存がある → 前提条件・制約

**L2 での簡略化の指針**:
- 現状の問題点: 表形式は維持、ただし項目数は最小限
- 既存パターン: 重要な判断のみ記載
- 代替案: 2案程度の比較で十分
- リスク: 実装上の注意点に絞る

> **注**: L2 は連番（1-8）で記述する。L3/L4 の全12セクション体系とは番号が異なるが、
> セクション名で対応関係は明確なため問題ない。

### L3/L4: Module/Project（完全版）

全12セクション + コンテキスト依存セクションを記述。

---

## 1. 設計方針

**目的**: この設計で何を重視するか、設計書全体のエグゼクティブサマリー

**記述内容:**
| 項目 | 説明 |
|------|------|
| 設計の核となる考え方 | 一言で言うとどういう設計か |
| 重視する価値 | シンプルさ、拡張性、パフォーマンス、etc |
| 採用するパターン/アプローチ | レイヤードアーキテクチャ、イベント駆動、etc |

**図の使用**: 不要

---

## 2. 現状の問題点

**目的**: なぜこの設計変更が必要なのか、変更する理由を定量的・具体的に示す

**記述内容:**
| 項目 | 説明 |
|------|------|
| 問題箇所の特定 | どこに問題があるか（ファイル名、関数名） |
| 定量的な根拠 | 行数、複雑度、重複箇所の数など |
| 問題の性質 | 責務の混在、密結合、テスト困難、etc |

**図の使用**: 不要。表形式での整理が効果的。

---

## 3. アーキテクチャ概要

**目的**: 設計全体の構造を可視化し、各要素の関係性を一目で把握できるようにする

**記述内容:**
| 項目 | 説明 |
|------|------|
| 全体構成図 | レイヤー/コンポーネント/モジュールの関係 |
| 依存関係 | 何が何に依存するか、依存の方向 |
| データ/処理の流れ | 主要なフローの概要 |

**L3/L4 での記述方法（重要）**:

L3/L4 でアーキテクチャに変更がある場合、以下の3部構成で記述する：

### 1. 現状のアーキテクチャ
- 既存の `architecture.md` の内容を引用または要約
- 現在の問題点を明確にする

### 2. 提案するアーキテクチャ
- **変更後の完全なアーキテクチャ**を記述（差分だけでなく全体像）
- 図を使って可視化
- 各要素の責務と関係性を明示

### 3. 差分と変更理由
| 変更箇所 | 現状 | 提案 | 理由 |
|---------|------|------|------|

この構成により：
- 設計書だけで全体像が把握できる（自己完結性）
- 差分が明確になる（レビュー可能性）
- 人間またはAI Agentが妥当性を検証できる
- `architecture.md` の更新は実装完了後に設計書の内容を反映

**コンテキスト別の図の形式:**
| コンテキスト | 適切な図 |
|-------------|---------|
| レイヤードアプリ | レイヤー図 + パッケージ依存図 |
| Terraform/IaC | リソース依存図、モジュール構成図 |
| マイクロサービス | サービス間通信図 |
| CLI/スクリプト | 処理フロー図 |
| 既存コード改修 | Before/After の構成比較 |

**図の使用**: **必須**。Mermaid `graph TD` またはASCII図で構成を可視化。

---

## 4. 既存パターンとの整合性

**目的**: 理想の設計と現実の落とし所を明示し、なぜその判断に至ったかを記録する

**記述内容:**
| 項目 | 説明 |
|------|------|
| 既存パターンの説明 | 現在のコードベースで採用されている慣習・パターン |
| あるべき姿（理想） | 制約がなければこうしたい、という設計 |
| 採用する設計（現実） | 既存パターンを考慮した落とし所 |
| 判断理由 | なぜ理想ではなく現実を選んだか |
| 将来の移行可能性 | 理想に近づける余地があるか（任意） |

**図の使用**: 不要。表形式で整理すると分かりやすい。

---

## 5. 代替案とトレードオフ

**目的**: 検討した選択肢を示し、なぜ現在の設計を選んだかを明確にする

**記述内容:**
| 項目 | 説明 |
|------|------|
| 検討した選択肢 | 2〜3つの代替案 |
| 各案のメリット/デメリット | 比較軸を揃えて評価 |
| 選定理由 | なぜその案を選んだか |

**図の使用**: 不要。表形式での比較が効果的。

---

## 6. リスク・注意点

**目的**: 実装時にハマりやすいポイント、事前に把握しておくべき落とし穴を明示する

**記述内容:**
| 項目 | 説明 |
|------|------|
| 技術的リスク | 実装難易度が高い箇所、未知の技術要素 |
| 互換性リスク | 既存機能への影響、破壊的変更 |
| 注意すべき実装ポイント | 見落としやすい考慮事項 |
| 軽減策 | リスクへの対処方法 |

**図の使用**: 不要。

---

## 7. スコープ外の明示

**目的**: 今回の設計で「やらないこと」を明確にし、スコープクリープを防ぐ

**記述内容:**
| 項目 | 説明 |
|------|------|
| 対象外の機能 | 今回は実装しない機能・変更 |
| 対象外の理由 | なぜ対象外としたか |
| 将来の対応予定 | 別チケット化するか、やらないか |

**図の使用**: 不要。

---

## 8. 前提条件・制約

**目的**: 設計が成り立つための前提、および外部要因による制約を明示する

**記述内容:**
| 項目 | 説明 |
|------|------|
| 技術的前提 | 依存ライブラリ、バージョン要件 |
| 環境前提 | OS、ランタイム、インフラ要件 |
| 外部制約 | 外部APIの制限、ライセンス制約 |
| プロジェクト制約 | 期限、リソース、互換性要件 |

**図の使用**: 不要。

---

## 9. ディレクトリ構造

**目的**: 物理的なファイル配置を明示し、どこに何があるかを把握できるようにする

**記述内容:**
| 項目 | 説明 |
|------|------|
| 変更後のディレクトリ構造 | ツリー形式での表示 |
| 新規追加ディレクトリの説明 | 各ディレクトリの責務 |

**図の使用**: 不要。ツリー形式のテキストで表現。

---

## 10. ファイル変更一覧

**目的**: 実装者が最初に確認すべき、具体的な変更対象を明示する

**記述内容:**
| 項目 | 説明 |
|------|------|
| 新規作成ファイル | パス + 簡潔な説明 |
| 変更ファイル | パス + 変更内容 |
| 削除ファイル | パス + 削除理由 |

**図の使用**: 不要。表形式が効果的。

---

## 11. 移行計画

**目的**: 実装の順序と依存関係を明示し、段階的に進められるようにする

**記述内容:**
| 項目 | 説明 |
|------|------|
| ステップ分割 | 論理的な実装順序 |
| 各ステップの内容 | 何を実装するか |
| 依存関係 | ステップ間の前後関係 |
| 完了条件（Definition of Done） | 各ステップの完了判断基準 |

**図の使用**: 任意。複雑な依存関係がある場合はMermaid Ganttチャートが有効。

---

## 12. テスト・動作確認方法

**目的**: 設計通りに実装されたことを確認する方法を明示する

**記述内容:**
| 項目 | 説明 |
|------|------|
| 自動テスト | テストファイル、テストケース |
| 手動確認手順 | 動作確認のシナリオ |
| 確認すべきポイント | 見落としやすい確認項目 |
| 完了判断基準 | 全体としての完了条件 |

**図の使用**: 不要。

---

## コンテキスト依存セクション

### ドメインモデル設計

**追加条件**: 状態を持つエンティティが存在する場合

**記述内容:**
- エンティティ一覧とその責務
- 状態遷移図（Mermaid `stateDiagram-v2`）
- クラス図（Mermaid `classDiagram`）
- 現在からの変更点

### ユースケース設計

**追加条件**: ビジネスロジックが存在する場合

**記述内容:**
- ユースケース一覧と責務
- 主要ユースケースのシーケンス図（Mermaid `sequenceDiagram`）
- 入出力の定義

### イベント設計

**追加条件**: イベント駆動/非同期処理が存在する場合

**記述内容:**
- イベント一覧
- イベントの発行元と購読先
- イベントのペイロード定義

### API設計

**追加条件**: REST API/GraphQL等のAPI変更がある場合

**記述内容:**
- エンドポイント一覧
- リクエスト/レスポンス形式
- エラーレスポンス定義

### データベース設計

**追加条件**: スキーマ変更がある場合

**記述内容:**
- テーブル定義（CREATE TABLE文またはER図）
- マイグレーション手順
- データ移行計画
