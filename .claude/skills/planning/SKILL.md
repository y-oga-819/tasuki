# 設計フェーズ開始

設計フェーズを開始し、タスクの規模に応じた設計ドキュメントを作成する。

## 設計書の目的

**実装フェーズで迷いが出ない設計書**を作成する。

- 設計方針を見るだけで全体像が把握できる
- 何を変更し、何を変更しないかが明確
- 実装の順序と完了条件が定義されている

小規模なタスクでも、手戻りコストを考慮すると詳細な設計が効率的である。

## 必須参照ファイル

設計書を作成する際は、以下のファイルを必ず参照すること：

- @sections.md - 各セクションの詳細な書き方
- @examples.md - 良い例・悪い例
- @template.md - 設計書テンプレート

## 実行手順

### Step 1: Worktree作成

> **前提条件**: `git-wt` がインストールされていること

1. 現在のブランチを確認する
2. mainブランチにいる場合：
   - `git wt feature/{feature-name}` を実行してworktreeを作成・移動

### Step 2: プロジェクト名と連番の決定

1. **プロジェクト名の決定**: カレントディレクトリの Git リポジトリ名（`basename $(git rev-parse --show-toplevel)`）を使用する。Git リポジトリでない場合はカレントディレクトリ名を使用する。
2. **連番の決定**: `~/.claude/designs/{project-name}/` ディレクトリ内の既存ファイルを確認し、最大の連番 + 1 を採番する（例: `0001`, `0002`, ...）。ファイルが存在しない場合は `0001` から開始する。
3. **出力先**: `~/.claude/designs/{project-name}/NNNN_{feature-name}.md`

### Step 3: タスクの理解と設計レベルの判定

1. ユーザーから依頼されたタスクの内容を確認する
2. 既存のコードベースを分析する
3. 以下の基準で設計レベルを判定する：

| レベル | 判断基準 | 対応 |
|--------|----------|------|
| L1: Patch | 影響範囲が1-2ファイル、既存の設計を変更しない | 設計書不要 |
| L2: Feature | 新しい振る舞いの追加、複数ファイルにまたがる変更 | 簡易版（8セクション） |
| L3: Module | 新しいディレクトリ/パッケージの作成、外部依存の追加 | 完全版（12セクション） |
| L4: Project | ゼロからのプロジェクト構築 | 完全版 + 全体設計 |

4. 判定結果をユーザーに報告し、確認を取る

**L2 での柔軟な対応**:

L2（Feature）レベルでは、以下のセクションを**必須**とし、残りは状況に応じて追加する：

**必須セクション（8個、連番で記述）**:
1. 設計方針
2. 現状の問題点（簡略化可）
3. アーキテクチャ概要
4. 既存パターンとの整合性
5. ディレクトリ構造
6. ファイル変更一覧
7. 移行計画
8. テスト・動作確認方法

**状況に応じて追加（9以降の番号で追加）**:
- 代替案とトレードオフ（選択肢がある場合）
- リスク・注意点（技術的難易度が高い場合）
- スコープ外の明示（スコープが曖昧な場合）
- 前提条件・制約（外部依存がある場合）

### Step 4: 設計ドキュメントの作成

設計レベルに応じて以下のドキュメントを作成する。

#### L1: Patch
- 設計ドキュメント作成は不要
- 変更内容をユーザーに説明し、実装に移行

#### L2: Feature（簡易版）
- `~/.claude/designs/{project-name}/NNNN_{feature-name}.md` を作成
- 必須8セクション + 必要に応じて追加セクション
- 手戻りを防ぐため、代替案やリスクも可能な限り記載する

#### L3/L4: Module/Project（完全版）
- L3: `~/.claude/designs/{project-name}/NNNN_{feature-name}.md` を作成
  - アーキテクチャセクションで「現状・提案・差分」を記述
  - 設計書が自己完結し、妥当性検証が可能になる
  - `architecture.md` の更新は実装完了後に行う
- L4: `~/.claude/designs/{project-name}/` 配下に全ドキュメント作成（NNNN_overview.md, NNNN_architecture.md 等）
- 12セクション全て + コンテキスト依存セクション

## 設計書の構成

### 普遍的セクション（必須）

全ての設計レベル（L2以上）で記述する。

| # | セクション | 目的 |
|---|-----------|------|
| 1 | 設計方針 | 設計書全体のエグゼクティブサマリー |
| 2 | 現状の問題点 | 変更する理由を定量的・具体的に示す |
| 3 | アーキテクチャ概要 | 全体構造の可視化 |
| 4 | 既存パターンとの整合性 | 理想と現実の落とし所を明示 |
| 5 | 代替案とトレードオフ | なぜこの設計を選んだか |
| 6 | リスク・注意点 | 実装時のハマりポイント |
| 7 | スコープ外の明示 | 今回やらないこと |
| 8 | 前提条件・制約 | 設計が成り立つための前提 |
| 9 | ディレクトリ構造 | 物理的なファイル配置 |
| 10 | ファイル変更一覧 | 新規/変更/削除ファイル |
| 11 | 移行計画 | 実装の順序と依存関係 |
| 12 | テスト・動作確認方法 | 完了条件と確認方法 |

### コンテキスト依存セクション（任意）

プロジェクトの特性に応じて追加する。

| セクション | 追加条件 |
|-----------|---------|
| ドメインモデル設計 | 状態を持つエンティティが存在する場合 |
| ユースケース設計 | ビジネスロジックが存在する場合 |
| イベント設計 | イベント駆動/非同期処理が存在する場合 |
| API設計 | REST API/GraphQL等のAPI変更がある場合 |
| データベース設計 | スキーマ変更がある場合 |

## 図の作成

必要に応じてMermaid図を作成する：

| 図の種類 | 用途 | Mermaid記法 |
|---------|------|-------------|
| 構成図 | アーキテクチャ概要 | `graph TD` |
| 状態遷移図 | ドメインモデル | `stateDiagram-v2` |
| シーケンス図 | ユースケース、処理フロー | `sequenceDiagram` |
| クラス図 | ドメインモデル、イベント | `classDiagram` |
| ガントチャート | 移行計画 | `gantt` |
| ER図 | データベース設計 | `erDiagram` |

## Step 5: ADRの確認

設計過程で以下の判断があった場合はADRを作成する：
- 技術選定
- 既存パターンからの逸脱
- トレードオフを伴う判断

`/adr` スキルを使用してADRを作成する。

---

## Step 6: 設計完了と次のステップ

次のステップを案内する：

> 設計が完了しました。
>
> **次のステップ（推奨順）**:
>
> 1. **`/planning-review`** で設計書の妥当性を検証（推奨）
>    - Subagent、人間、Codex の複数の視点から品質を保証
>    - 整合性、完全性、実現可能性をチェック
>
> 2. **`/plan-tasks`** でタスクリストを生成（推奨）
>    - 設計書の移行計画を実装可能な粒度に分解
>    - Phase/Targets/References/Tasks 形式で出力
>    - `/clear` 後も再開可能な永続化されたタスクリスト
>
> 3. **`/implement`** で実装を開始
>    - タスクリストがあれば読み込んで Phase 単位で実装
>    - タスクリストがなければ設計書から直接実装
>
> **コンテキストをリフレッシュする場合**:
> `/clear` を実行してから `/implement` を呼び出してください。
> タスクリストファイルがあれば、自動的に読み込まれます。
