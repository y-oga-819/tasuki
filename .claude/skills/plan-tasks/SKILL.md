# タスクリスト生成

設計書を読み込み、実装可能な粒度のタスクリストをMarkdownファイルとして生成する。

## 目的

**設計書の移行計画を、実装フェーズで迷わないタスクリストに変換する**

- Phase/Targets/References/Tasks の構造で整理
- TDDの原則に沿ったタスク順序
- `/clear` 後も再開可能な永続化されたタスクリスト

## 必須参照ファイル

タスクリストを作成する際は、以下のファイルを必ず参照すること：

- @template.md - タスクリストのテンプレート

---

## 実行手順

### Step 1: 設計書の読み込み

以下の優先順位で設計ドキュメントを読み込む：

1. `~/.claude/designs/{project-name}/NNNN_{対象機能}.md`（指定された場合）
2. `~/.claude/designs/{project-name}/` 内の最新の設計書（連番が最大のもの）
3. ユーザーが指定したファイル

> **プロジェクト名の決定**: Gitリポジトリ名（`basename $(git rev-parse --show-toplevel)`）またはカレントディレクトリ名を使用する。

設計書から以下のセクションを特に注目して読み込む：
- **10. ファイル変更一覧** - 変更対象ファイルの特定
- **11. 移行計画** - Step の抽出
- **12. テスト・動作確認方法** - 完了条件の参照

### Step 2: タスクリストの構造化

設計書の移行計画を、以下の構造に変換する：

```markdown
## Phase N: {Phase名}

{Phaseの目的と概要}

### Targets

- {変更対象ファイル} - {変更内容の要約}

### References

- {参照ファイル} - {参照理由}

### Tasks

- [ ] {タスク1}
- [ ] {タスク2}
```

#### 変換ルール

**Phase の決定:**
- 設計書の Step をそのまま Phase に対応させる
- または、論理的な作業単位（コミット単位）で分割

**Targets の抽出:**
- 設計書「10. ファイル変更一覧」から、該当 Phase で変更するファイルを抽出
- 新規作成・変更・削除を明記

**References の抽出:**
- 該当 Phase で参照が必要なファイル（変更はしない）
- 既存パターンの確認、移植元のコード等

**Tasks の分解:**
- TDDの原則に沿って分解：
  1. テストを書く
  2. 実装する（仮実装 → 本実装）
  3. リファクタリング（必要な場合）
- 1タスク = 小さな変更単位（数十行程度）
- タスクはネストしない（フラットなリスト）

### Step 3: タスクリストの出力

タスクリストを以下のパスに出力する：

```
~/.claude/designs/{project-name}/tasks/NNNN_{feature-name}-tasks.md
```

**ディレクトリ:** `tasks/` ディレクトリが存在しない場合は作成する。

**ファイル名の決定:**
- 設計書が `NNNN_{feature-name}.md` の場合 → 同じ連番で `NNNN_{feature-name}-tasks.md`

### Step 4: 完了報告

ユーザーに以下を報告：

1. 作成したタスクリストのパス
2. Phase 数とタスク数の概要
3. 次のステップの案内

> タスクリストを作成しました。
>
> **出力先:** `~/.claude/designs/{project-name}/tasks/NNNN_{feature-name}-tasks.md`
>
> **概要:**
> - Phase 数: {N}
> - タスク数: {M}
>
> **次のステップ:**
> 1. タスクリストの内容を確認・修正
> 2. `/implement` で実装を開始

## タスク分解の判断基準

### 粒度の目安

| 粒度 | 例 | 判断 |
|------|-----|------|
| 大きすぎる | 「ユースケース層を実装」 | 分割が必要 |
| 適切 | 「CreateTaskUseCase のインターフェースを定義」 | OK |
| 小さすぎる | 「import 文を追加」 | 他と統合 |

**判断のポイント:**
- 1タスク完了で意味のある進捗になるか
- テストで完了を確認できるか
- 他のタスクと独立して実行できるか

> 出力形式や記述ルールの詳細は @template.md を参照

---

## 注意事項

- 設計書の移行計画が抽象的な場合は、コードベースを分析して具体化する
- タスクの順序は依存関係を考慮する（先に作るものが先）
- 各タスクは独立して完了判定できる粒度にする
- 行番号は記載しない（ファイル名とシンボルのみ）
