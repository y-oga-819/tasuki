## ユースケース 1: Claude Code の変更をレビューする（基本フロー）

**シナリオ**: Claude Code にコード変更を依頼した後、その変更内容をレビューしてフィードバックを返す。

### ステップ

```
1. Claude Code がコード変更を完了
        ↓
2. ターミナルで tasuki を起動
        ↓
3. 差分を確認し、行レベルでコメントを追加
        ↓
4.「Copy All」でフィードバックをコピー
        ↓
5. Claude Code にペーストしてフィードバック
```

### 操作手順

**1) tasuki を起動する**

```bash
# プロジェクトのルートディレクトリで実行
tasuki
```

アプリが起動し、未コミットの変更（staged + working tree）がすべて表示されます。

**2) ファイルを選択して差分を確認する**

- 左サイドバーにファイル一覧が表示される
- 各ファイル名の横に `+追加行数 -削除行数` が表示される
- ファイルをクリックすると、メイン画面に差分が表示される
- lock ファイル等の自動生成ファイルはデフォルトで折りたたまれている

**3) レビューコメントを追加する**

- 差分の行番号をクリックすると、その行にコメントを追加できる
- 複数行を範囲選択してコメントすることも可能
- コメントタイプを選択可能: `comment` / `suggestion` / `question` / `approval`

**4) 判定を下してフィードバックをコピーする**

- 画面下部の ReviewPanel で「Approve」または「Request Changes」を選択
- 「Copy All」ボタンをクリック
- 以下のような構造化テキストがクリップボードにコピーされる:

```markdown
## Review Result: Request Changes

### src/components/DiffViewer.tsx
- L42-L45
  > const handleClick = () => {
  >   setSelected(true);
  > };
  この処理はメモ化した方がパフォーマンスが改善されるのでは？

- L78
  > const items = data.filter(x => x.active);
  変数名 x を item に変更してください。

### src/utils/format-review.ts
- L15-L20
  > const verdictLabel =
  >   verdict === "approve"
  >     ? "Approve"
  >     : verdict === "request_changes"
  verdict が null の場合のハンドリングが必要です。

### Summary
Please address the above comments and request re-review.
```

**5) Claude Code にフィードバックをペーストする**

コピーした内容を Claude Code のプロンプトにそのまま貼り付けると、ファイルパス・行番号・コードスニペット付きの構造化フィードバックとして AI が正確に理解し、修正を実行してくれます。

## ユースケース 2: ステージング済み変更だけをレビューする

**シナリオ**: `git add` でステージングした変更だけを確認したい。作業中の未ステージ変更はノイズになるため除外する。

### 操作手順

```bash
# ステージング済み変更のみ表示
tasuki staged
```

または、未ステージの変更のみを見たい場合:

```bash
# ワーキングツリーの変更のみ表示
tasuki working
```

### 使い分け

| コマンド | 対象 | ユースケース |
|---------|------|-------------|
| `tasuki` | staged + working | コミット前に全変更を確認 |
| `tasuki staged` | staged のみ | コミット対象だけを最終確認 |
| `tasuki working` | working のみ | 作業途中の変更を確認 |

## ユースケース 3: ブランチ間の差分を比較する

**シナリオ**: feature ブランチで作業した全変更を、main ブランチと比較してレビューする。

### 操作手順

```bash
# feature ブランチと main ブランチの差分を比較
tasuki feature main
```

これにより、`feature` ブランチと `main` ブランチ間のすべての変更が表示されます。

### 具体例

```bash
# develop ブランチ vs main
tasuki develop main

# リリースブランチとステージング環境の比較
tasuki release/v2.0 staging

# リモートブランチとの比較
tasuki origin/feature-auth origin/main
```

## ユースケース 4: 特定コミットの変更を確認する

**シナリオ**: 過去のコミットの変更内容を個別に確認したい。

### 操作手順

```bash
# コミットハッシュを指定
tasuki abc1234

# HEAD から N コミット前の差分
tasuki HEAD~3

# 直前のコミット
tasuki HEAD~1
```

コミットとその親コミットの差分が表示されます。

## ユースケース 5: 設計書を閲覧する

**シナリオ**: `docs/` ディレクトリにある設計書を Mermaid ダイアグラム付きで閲覧する。

### 操作手順

```bash
# docs/ 以下の全 Markdown ファイルを一覧表示
tasuki docs

# 特定の設計書を直接開く
tasuki docs architecture.md
```

### 設計書ビューアの機能

- Markdown が整形表示される（GitHub Flavored Markdown 対応）
- 見出しが 4 つ以上あれば TOC（目次）が自動生成される
- Mermaid 記法のコードブロックがダイアグラムとして描画される
- コードブロックにはシンタックスハイライトが適用される

### 設計書へのコメント

設計書のセクションにもコメントを付けることが可能です。コメントは「Copy All」でコードコメントと一緒に出力されます。

```markdown
## Review Result: Comments

### docs/design.md
- アーキテクチャ: バックエンドのエラーハンドリング方針を追記してほしい
- データモデル: ユーザーテーブルの外部キー制約が図に含まれていない
```

## ユースケース 6: 設計書を参照しながらコードレビューする

**シナリオ**: 設計書の要件とコードの実装を並べて確認し、設計通りに実装されているか検証する。

### 操作手順

**1) tasuki を起動する**

```bash
tasuki
```

**2) 表示モードを「Diff + Docs」に切り替える**

- ツールバーで「Diff + Docs」タブをクリック
- 画面が左右に分割され、左に差分ビューア、右に設計書が表示される

**3) 設計書を選択する**

- 左サイドバーの「Docs」セクションから目的の設計書をクリック
- 右ペインに設計書が表示される

**4) コードと設計書を見比べながらコメントする**

- 左ペインの差分で行番号をクリックしてコードにコメント
- 右ペインの設計書のセクションにもコメント可能
- 両方のコメントが「Copy All」でまとめて出力される

### レイアウト切り替え

差分ビューアのレイアウトも状況に応じて切り替え可能:

| レイアウト | 説明 | 適した状況 |
|-----------|------|-----------|
| Split | 変更前/変更後を左右に並べて表示 | 行単位の変更を細かく比較する場合 |
| Unified | 変更前/変更後を上下に統合して表示 | ファイル全体の流れを追う場合 |

長い行がある場合の表示オプション:

| オプション | 説明 |
|-----------|------|
| Scroll | 長い行は水平スクロールで表示 |
| Wrap | 長い行を折り返して表示 |

## ユースケース 7: AI との反復レビューサイクル

**シナリオ**: Claude Code と複数回のやり取りを繰り返し、品質を段階的に向上させる。

### 反復サイクルのフロー

```
┌───────────────────────────────────────────────────┐
│                                                     │
│   Claude Code        Tasuki          人間           │
│   ──────────        ──────          ────           │
│                                                     │
│   ① 機能を実装  ──→  ② 差分を表示  ──→  ③ レビュー  │
│                                                     │
│   ⑥ 修正を実施  ←──  ⑤ Copy All    ←──  ④ コメント  │
│                                                     │
│   ⑦ 修正完了    ──→  ⑧ 差分を更新  ──→  ⑨ 再レビュー │
│                                       │             │
│                              OK ←─────┘             │
│                               │                     │
│                          ⑩ Approve                  │
│                                                     │
└───────────────────────────────────────────────────┘
```

### 具体的な手順

**ラウンド 1: 初回レビュー**

```bash
# 1. Claude Code に機能実装を依頼（例: ユーザー認証機能）
# → Claude Code がコードを生成

# 2. tasuki でレビュー開始
tasuki

# 3. 差分を確認し、以下のようなコメントを追加:
#    - src/auth.ts:L15 「パスワードのハッシュ化が必要」
#    - src/auth.ts:L32 「トークンの有効期限を設定すべき」
#    - src/routes.ts:L8  「エラーレスポンスの形式を統一して」

# 4. 「Request Changes」→「Copy All」→ Claude Code にペースト
```

**ラウンド 2: 修正後の再レビュー**

```bash
# 5. Claude Code が修正を実施
# → tasuki のファイル監視が変更を自動検知し、差分が更新される

# 6. 更新された差分を確認
#    - 前回のコメントが対応されているか確認
#    - 新たな問題がないか確認

# 7. 問題なければ「Approve」→「Copy All」→ Claude Code にペースト
```

### Copy All の出力例（Approve 時）

```markdown
## Review Result: Approve

### Summary
LGTM. Changes look good.
```

## ユースケース 8: リアルタイムでファイル変更を監視しながらレビューする

**シナリオ**: Claude Code が作業中の変更をリアルタイムで監視し、変更が入るたびに差分を確認する。

### 仕組み

Tasuki はファイルシステムの変更を自動監視しています:

- ファイルが変更されると 500ms のデバウンス後に差分が自動更新される
- 以下のディレクトリは監視対象外:
  - `.git/`, `node_modules/`, `target/`, `dist/`
  - `.tasuki/reviews/`, `__pycache__/`, `.next/`, `.nuxt/`

### 操作手順

```bash
# 1. tasuki を起動したまま
tasuki

# 2. 別のターミナルで Claude Code がコードを変更
#    → tasuki の画面が自動的に更新される

# 3. 変更が入るたびにリアルタイムで差分を確認できる
```

手動でリロードする必要はありません。ファイルが保存されるたびに最新の差分が反映されます。

---

## 補足: 自動生成ファイルの折りたたみ

以下のファイルは自動的に折りたたまれた状態で表示されます（クリックで展開可能）:

- **lock ファイル**: `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`, `Cargo.lock`, `Gemfile.lock`, `poetry.lock`, `composer.lock`
- **minified ファイル**: `*.min.js`, `*.min.css`
- **ソースマップ**: `*.map`
- **自動生成コード**: `.generated.` や `__generated__` を含むパス

これにより、レビュー対象のコードに集中しやすくなっています。

## コマンド早見表

| コマンド | 説明 |
|---------|------|
| `tasuki` | 全未コミット変更（staged + working）を表示 |
| `tasuki .` | 同上 |
| `tasuki staged` | ステージング済み変更のみ表示 |
| `tasuki working` | 未ステージ変更のみ表示 |
| `tasuki HEAD~N` | N コミット前との差分を表示 |
| `tasuki <commit>` | 特定コミットの差分（vs 親コミット）を表示 |
| `tasuki <ref1> <ref2>` | 2 つの ref 間の差分を表示 |
| `tasuki docs` | docs/ 以下のドキュメント一覧を表示 |
| `tasuki docs <file>` | 特定のドキュメントを表示 |

## UI 操作早見表

| 操作 | 説明 |
|------|------|
| ファイルクリック（サイドバー） | 対象ファイルの差分を表示 |
| 行番号クリック（差分ビューア） | その行にコメントを追加 |
| 行範囲選択（差分ビューア） | 範囲を指定してコメントを追加 |
| Docs / Diff / Diff+Docs（ツールバー） | 表示モードを切り替え |
| Split / Unified（ツールバー） | 差分のレイアウトを切り替え |
| Scroll / Wrap（ツールバー） | 長い行の表示方法を切り替え |
| Approve / Request Changes（ReviewPanel） | レビュー判定を設定 |
| Copy All（ReviewPanel） | 全コメントを構造化テキストとしてコピー |
| Copy（個別コメント） | 単一コメントを `file:L行番号` 形式でコピー |
