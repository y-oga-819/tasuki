# AI協業ワークフロー最適化 設計書

## 1. 現状分析

### 1.1 現在のワークフロー

```
設計skill → 設計書初稿 → Tasukiでレビュー → context clear
→ タスク分解skill → context clear
→ 実装skill → Tasukiでレビュー → commit → (繰り返し)
→ PR作成 → context clear
→ レビューskill → 修正 → Approve → マージ
```

### 1.2 構造的問題の特定

#### 問題1: Context Clearが「情報の断崖」を作る

Context clearは4回発生する（設計後、タスク分解後、実装完了後、PRレビュー前）。各clearは必要な措置だが、暗黙知の断絶を引き起こす。

**失われる情報の例:**
- 設計時に検討した代替案とその棄却理由
- 「ここは意図的にこう書いた」という設計判断の文脈
- レビューで議論した結果の背景
- 実装中に発見した設計の問題とその場での対処

現状、これらは次のフェーズのAIに引き継がれない。新しいコンテキストのAIは設計書のテキストだけを読み、**なぜその設計になったか**を知らないまま作業する。

#### 問題2: フェーズ間のインターフェースが未定義

各skillが何を入力として受け取り、何を出力するかが暗黙的。人間の頭の中でフローが繋がっている。

```
設計skill → ???（設計書.md）→ タスク分解skill → ???（タスクリスト）→ 実装skill
```

「???」の部分が標準化されていないため:
- 設計書のフォーマットが毎回異なる
- タスク分解の粒度が一定しない
- 実装skillが必要とする情報が欠けることがある

#### 問題3: レビューが「ゲート」であって「協業」ではない

現在のフローでは、レビューは「通過/拒否」の判定ポイントとして機能している。しかし本来のレビューは:
- 設計意図の確認と共有
- 見落としの発見
- 代替案の検討
- 知見の蓄積

これらの「対話的」な側面が、コピペ連携では十分に機能しない。

#### 問題4: 人間のレビュー負荷が高い

設計レビューとコードレビューの両方を人間が主導しており:
- レビュー観点が毎回手探り
- 疲労による品質低下
- 「何を見るべきか」の判断自体がコスト

#### 問題5: 線形フローが後戻りに弱い

実装中に設計の問題を発見した場合、フローを巻き戻す手段が定義されていない。結果として:
- その場しのぎの対処をするか
- 全体をやり直すか

の二択になりやすい。

## 2. パラダイムシフト: Tasukiを「開発セッションの状態マシン」にする

### 2.1 核心的な転換

**現状**: Tasuki = レビューツール（フローの中の1ステップ）
**目標**: Tasuki = 開発セッション全体の状態管理と品質保証のプラットフォーム

```
┌─────────────────────────────────────────────────────────────┐
│                    Tasuki (状態マシン)                        │
│                                                              │
│  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐  │
│  │ 設計 │───→│ 分解 │───→│ 実装 │───→│  PR  │───→│レビュー│  │
│  │      │←──│      │←──│      │←──│      │←──│      │  │
│  └──────┘    └──────┘    └──────┘    └──────┘    └──────┘  │
│       ↑           ↑           ↑           ↑           ↑      │
│       │           │           │           │           │      │
│  ┌────┴───────────┴───────────┴───────────┴───────────┴──┐  │
│  │           ハンドオフドキュメント (構造化JSON)             │  │
│  │         + 意思決定ログ + レビュー履歴                     │  │
│  └───────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
         ↕                    ↕                    ↕
    Claude Code            Codex             人間 (レビュアー)
```

### 2.2 フェーズの状態と遷移

```
                    ┌───────────────┐
                    │   セッション    │
                    │    開始        │
                    └───────┬───────┘
```

## 3. 具体的改善策

### 3.1 ハンドオフプロトコルの導入

Context clear前に、構造化された**ハンドオフドキュメント**を生成する。これはTasukiが管理し、次のフェーズのskillに自動的に渡される。

#### ハンドオフドキュメントの構造

```
.tasuki/sessions/{session_id}/
├── session.json              # セッション全体のメタデータ
├── design/
│   ├── spec.md               # 設計書本体
│   ├── decisions.json        # 意思決定ログ
│   └── handoff.json          # 設計→分解ハンドオフ
├── decomposition/
│   ├── steps.json            # タスク分解結果
│   └── handoff.json          # 分解→実装ハンドオフ
├── implementation/
│   ├── step-{n}/
│   │   ├── review.json       # ステップレビュー結果
│   │   └── changes.json      # 変更の要約
│   └── handoff.json          # 実装→PRハンドオフ
└── pr/
    ├── review.json            # PRレビュー結果
    └── history.json           # レビューラウンドの履歴
```

#### session.json

```json
{
  "id": "sess_20260214_feature_auth",
  "title": "ユーザー認証機能の実装",
  "created_at": "2026-02-14T10:00:00Z",
  "current_phase": "implementation",
  "current_step": 3,
  "total_steps": 7,
  "repository": "my-app",
  "branch": "feature/auth",
  "phases": {
    "design": { "status": "approved", "completed_at": "..." },
    "decomposition": { "status": "approved", "completed_at": "..." },
    "implementation": { "status": "in_progress", "started_at": "..." },
    "pr": { "status": "pending" },
    "review": { "status": "pending" }
  }
}
```

#### decisions.json (意思決定ログ)

```json
{
  "decisions": [
    {
      "id": "dec_001",
      "phase": "design",
      "question": "認証方式をJWTにするかセッションベースにするか",
      "decision": "JWT",
      "rationale": "SPAとの相性、マイクロサービス展開を見据えて",
      "alternatives_considered": [
        { "option": "セッションベース", "rejected_because": "スケールアウト時にセッションストアが必要" }
      ],
      "decided_by": "human",
      "timestamp": "2026-02-14T10:30:00Z"
    }
  ]
}
```

#### handoff.json (フェーズ間引き継ぎ)

```json
{
  "from_phase": "design",
  "to_phase": "decomposition",
  "summary": "JWT認証機能の設計が完了。以下の点に留意して分解すること。",
  "key_constraints": [
    "既存のUserモデルを拡張する（新規テーブルは作らない）",
    "リフレッシュトークンのローテーションは必須"
  ],
  "unresolved_items": [
    "パスワードリセットフローのUI設計は後回し"
  ],
  "review_highlights": [
    {
      "comment": "エラーレスポンスの形式を統一すること",
      "file": "docs/design.md",
      "section": "API設計"
    }
  ],
  "context_for_next_agent": "設計書の4.2節「エッジケース」を必ず読んでから分解を開始すること"
}
```

**効果**: AIエージェントがcontext clear後も、前フェーズの重要な文脈を構造化された形で受け取れる。人間の「頭の中のコンテキスト」を外部化する。

### 3.2 レビュー観点の構造化

レビュー時に「何を見るべきか」をフェーズごとに定義し、Tasukiが提示する。

#### 設計レビュー観点

```json
{
  "phase": "design",
  "perspectives": [
    {
      "category": "要件の充足",
      "checks": [
        "やりたいことが設計書に正確に反映されているか",
        "スコープ外と明示されている事項は妥当か"
      ]
    },
    {
      "category": "技術的妥当性",
      "checks": [
        "既存アーキテクチャとの整合性はあるか",
        "選定した技術に対して代替案は検討されているか",
        "パフォーマンス上の懸念はないか"
      ]
    },
    {
      "category": "実装可能性",
      "checks": [
        "曖昧な表現がなく、実装者が迷わないか",
        "エッジケースが考慮されているか",
        "テスト戦略が含まれているか"
      ]
    },
    {
      "category": "リスク",
      "checks": [
        "破壊的変更の有無と影響範囲",
        "セキュリティ上の考慮事項",
        "ロールバック可能性"
      ]
    }
  ]
}
```

#### コードレビュー観点（ステップレビュー用）

```json
{
  "phase": "step_review",
  "perspectives": [
    {
      "category": "設計書との一致",
      "checks": [
        "設計書の仕様通りに実装されているか",
        "勝手な判断で設計から逸脱していないか"
      ]
    },
    {
      "category": "コード品質",
      "checks": [

        "型安全性は保たれているか"
      ]
    },
    {
      "category": "テスト",
      "checks": [
        "変更に対するテストが書かれているか",
        "テストは正しいことをテストしているか"
      ]
    },
    {
      "category": "副作用",

```

**Tasuki UI上での表示**: レビュー開始時にチェックリストとして表示し、各項目に対してコメントや「確認済み」のマークを付けられる。レビューの網羅性を保証する。

### 3.3 マルチエージェント並列レビューの統合

設計レビューフェーズでClaude CodeやCodexが並列でレビューする場合、Tasukiがレビュー結果を統合する。

```
┌─────────┐   レビュー結果A    ┌──────────────┐
│Claude Code│──────────────→│              │
└─────────┘                │              │
                           │    Tasuki     │   統合ビュー
┌─────────┐   レビュー結果B  │  (レビュー統合) │──────────→ 人間
│  Codex   │──────────────→│              │
└─────────┘                │              │
                           │              │
┌─────────┐   手動レビュー  │              │
│  人間    │──────────────→│              │
└─────────┘                └──────────────┘
```

#### 統合ロジック

1. **同じ箇所への指摘の検出**: 複数エージェントが同じファイル・同じ行範囲にコメントした場合、グループ化して表示
2. **矛盾の検出**: エージェントAが「この実装でOK」、エージェントBが「ここは変えるべき」と言った場合、矛盾としてハイライト
3. **重要度の推定**: 複数エージェントが同じ指摘をした場合、重要度が高いとマーク

#### 実装方針

各エージェントのレビュー結果を統一フォーマットのJSONとして受け取り、Tasukiが `.tasuki/sessions/{id}/reviews/` にエージェント別に保存。UIでは統合ビューとエージェント別ビューを切り替えられる。

```json
{
  "agent": "claude-code",
  "timestamp": "2026-02-14T11:00:00Z",
  "comments": [
    {
      "file": "docs/design.md",
      "section": "API設計",
      "body": "エラーレスポンスの形式が統一されていない",
      "severity": "warning",
      "suggestion": "RFC 7807 Problem Details形式を採用する"
    }
  ],
  "overall": "conditional_approve",
  "summary": "基本設計は問題ないが、エラーハンドリングの統一が必要"
}
```

### 3.4 設計変更のフィードバックループ

実装中に設計の問題を発見した場合の「後戻り」パスを正式に定義する。

#### 設計変更リクエスト (Design Change Request: DCR)

```json
{
  "id": "dcr_001",
  "requested_by": "implementation_agent",
  "phase": "implementation",
  "step": 3,
  "type": "design_amendment",
  "severity": "blocking",
  "description": "設計書ではUserテーブルにtokenカラムを追加する設計だが、既存のマイグレーションとの整合性が取れない",
  "proposed_change": "token_storeを別テーブルとして分離する",
  "impact": [
    "設計書 4.1 節の修正",
    "ステップ3-5の再分解が必要"
  ],
  "status": "pending_review"
}
```

#### フロー

```
実装中に問題発見 → DCR作成 → Tasukiに表示
                                ↓
                         人間がDCRを判断
                          ├── 承認 → 設計書修正 → 影響するステップの再分解
                          ├── 棄却 → 実装側で別の方法を検討
                          └── 保留 → 現ステップを完了してから再議論
```

**Tasuki上でのDCR管理**: 設計書ビューにDCRバッジを表示し、該当箇所をハイライト。人間がDCRの判断を行い、その結果をdecisions.jsonに記録する。

### 3.5 スキル入出力の標準化

各Claude Code skillが何を入力として受け取り、何を成果物として出力するかを厳密に定義する。

#### 設計 skill

```
入力:
  - 人間の要求（自然言語）
  - 既存の設計書・アーキテクチャ図（リポジトリのdocs/配下）
  - CLAUDE.md（プロジェクトのコンテキスト）

出力:
  - docs/{feature_name}/design.md（設計書本体）
  - .tasuki/sessions/{id}/design/decisions.json（意思決定ログ）
  - .tasuki/sessions/{id}/design/handoff.json（次フェーズへの引き継ぎ）

完了条件:
  - Tasukiでの設計レビューがApproveされている
```

#### タスク分解 skill

```
入力:
  - 設計書（design.md）
  - ハンドオフドキュメント（handoff.json）
  - 意思決定ログ（decisions.json）

出力:
  - .tasuki/sessions/{id}/decomposition/steps.json
  - 各ステップの実装手順詳細
  - .tasuki/sessions/{id}/decomposition/handoff.json

完了条件:
  - Tasukiでのタスク分解レビューがApproveされている

steps.json の構造:
  {
    "steps": [
      {
        "number": 1,
        "title": "Userモデルの拡張",
        "description": "...",
        "files_to_modify": ["src/models/user.ts", "src/db/migrations/..."],
        "files_to_create": ["src/models/token.ts"],
        "acceptance_criteria": [
          "Userモデルにpassword_hashフィールドが追加されている",
          "マイグレーションが正常に実行できる"
        ],
        "dependencies": [],
        "estimated_complexity": "low"
      }
    ]
  }
```

#### 実装 skill

```
入力:
  - 設計書（design.md）
  - タスク分解結果（steps.json）から対象ステップ
  - ハンドオフドキュメント（handoff.json）
  - 前ステップのレビュー結果（あれば）

出力:
  - コード変更（git diff）
  - .tasuki/sessions/{id}/implementation/step-{n}/changes.json

完了条件:
  - コード変更がTasukiでレビューされApproveされている
  - ステップのacceptance_criteriaを満たしている
```

#### レビュー skill

```
入力:
  - PR差分
  - 設計書（design.md）
  - 意思決定ログ（decisions.json）
  - 各ステップのレビュー履歴

出力:
  - PRへのレビューコメント
  - Approve / Request Changes

完了条件:
  - 全指摘が解決されApproveされている
```

### 3.6 人間の行動と意識の変革

#### 3.6.1 「AIの作業を監督する」から「AIと交互に判断する」へ

現在の意識:
```
人間: 指示を出す → AIが作業する → 人間が結果を検査する → やり直し or 承認
```

目指す意識:
```
人間: 意図を伝える → AIが構造化する → 人間が判断する → AIが具体化する → 人間が判断する → ...
```

具体的な変化:
- **設計フェーズ**: AIに「設計して」ではなく「選択肢を出して」と指示する。AIが3つの設計案を構造化して提示し、人間がトレードオフを見て判断する
- **レビューフェーズ**: 「直して」ではなく「なぜこうした？」と聞く。AIの意図を理解した上で判断する
- **DCR**: 「設計通りにやれ」ではなく、実装中の発見を設計へのフィードバックとして活かす

#### 3.6.2 レビュー時間の使い方

**やめるべきこと**:
- typo、フォーマット、lintエラーの指摘 → 自動化で排除
- コード全行の読み込み → 変更の意図を理解してから差分を見る
- 一人で全観点をカバー → マルチエージェントレビューで分担

**注力すべきこと**:
- 設計書との整合性の確認
- ビジネスロジックの正しさ
- エッジケースの考慮
- 「これで本当に要件を満たすか？」

#### 3.6.3 意思決定の外部化

人間の頭の中にある「なぜそう決めたか」を必ずTasukiに記録する習慣をつける。

記録すべきもの:
- 設計の分岐点での判断理由
- 「今はやらない」と決めた事項
- レビューで見逃した（後で気づいた）ポイント
- 「次回からはこう指示した方がいい」という気づき

→ これらが蓄積されることで、skill側のプロンプトが改善され、フロー全体の品質が向上する。

### 3.7 ワークフロー自動化の段階

#### Phase A: 基盤構築（セッション管理 + ハンドオフ）

Tasukiの機能追加:
1. **セッション管理UI** — 現在のフェーズ、進捗を可視化
2. **ハンドオフドキュメントの表示** — フェーズ遷移時に引き継ぎ内容を確認
3. **意思決定ログUI** — 設計判断を記録・閲覧
4. **レビュー観点チェックリスト** — フェーズごとの観点を表示

Claude Code側:
1. 各skillが `.tasuki/sessions/` にハンドオフドキュメントを書き出す
2. 各skillが起動時にハンドオフドキュメントを読み込む
3. CLAUDE.mdにセッション管理のプロトコルを記述

#### Phase B: レビュー品質の向上

Tasukiの機能追加:
1. **マルチエージェントレビュー統合** — 複数レビュー結果の統合表示
2. **DCR管理UI** — 設計変更リクエストの作成・承認フロー
3. **レビュー履歴の可視化** — 過去のレビューラウンドの参照
4. **レビュー観点のカスタマイズ** — プロジェクトごとの観点定義

Claude Code側:
1. レビューskillがTasukiの統一フォーマットで結果を出力
2. 実装skillがDCRを生成できる

#### Phase C: フロー最適化

Tasukiの機能追加:
1. **フェーズ遷移の自動化** — Approve時に次フェーズの準備を自動実行
2. **知見の蓄積** — レビューパターンの分析、頻出指摘の自動検出
3. **テンプレート管理** — 設計書・タスク分解のテンプレートをプロジェクトごとに管理
4. **ワークフローのカスタマイズ** — フェーズの追加・削除・順序変更

## 4. 最適化されたワークフロー（目標像）

### 4.1 全体フロー

```
人間: 「認証機能を追加したい」

  ↓ (1) Tasukiでセッション開始

Tasuki: セッション作成 → session.json生成
  → 既存の設計書・アーキテクチャ図をリスト表示
  → 人間が関連ドキュメントを選択

  ↓ (2) 設計skill起動（Claude Code）

Claude Code: 設計書初稿作成
  → 選択肢がある場合は構造化して提示
  → decisions.jsonに検討過程を記録
  → design.md + handoff.json を出力

  ↓ (3) Tasukiで設計レビュー

Tasuki: 設計書 + レビュー観点チェックリストを表示
  → 並行してClaude Code / Codexがレビュー（結果はTasukiに集約）
  → 人間がマルチエージェントレビュー結果 + 自分の視点でレビュー
  → Approve → handoff.json自動生成

  ↓ (4) context clear → タスク分解skill起動

Claude Code: handoff.json + decisions.json + design.md を読み込み
  → 実装ステップに分解（steps.json）
  → 各ステップのacceptance_criteriaを定義

  ↓ (5) Tasukiでタスク分解レビュー

Tasuki: steps.jsonをボード形式で表示
  → 粒度・依存関係・acceptance_criteriaの妥当性を確認
  → Approve → handoff.json自動生成

  ↓ (6) context clear → 実装skill起動（ステップごと）

Claude Code: ステップNの実装
  → 設計書 + handoff + ステップ定義に従って実装
  → 問題発見時はDCRを生成

  ↓ (7) Tasukiでステップレビュー

Tasuki: 差分表示 + 設計書サイドバイサイド + acceptance_criteriaチェックリスト
  → コードレビュー観点に沿ってレビュー
  → Approve → 次ステップへ / Reject → 修正依頼
  → DCRがあれば人間が判断

  ↓ (6)-(7) 全ステップ完了まで繰り返し

  ↓ (8) PR作成

Claude Code: PR作成（設計書・意思決定ログへのリンク付き）

  ↓ (9) context clear → レビューskill起動

Claude Code: PR全体を設計書と照合してレビュー
  → 個別ステップでは見えなかった全体の一貫性を確認
  → レビュー結果をTasukiに出力

  ↓ (10) Tasukiで最終レビュー

Tasuki: PRレビュー結果 + 人間の視点で最終確認
  → 修正があれば修正サイクル
  → Approve → 人間がマージ
```

## 8. まとめ: 3つの本質的な転換

1. **「レビューツール」から「開発セッション管理プラットフォーム」へ**
   - Tasukiが各フェーズの状態を管理し、フェーズ間のコンテキストを保持する
   - Context clearが「情報の断崖」ではなく「整理されたバトンタッチ」になる

2. **「人間がすべてレビューする」から「人間はAIレビューを判断する」へ**
   - マルチエージェントレビューで定型的な観点はAIがカバー
   - 人間は矛盾の解決、ビジネス判断、設計意図の確認に集中する

3. **「線形フロー」から「状態マシン + フィードバックループ」へ**
   - DCRによる公式な後戻りパス
   - 意思決定ログによる判断の蓄積と活用
   - 各フェーズが独立して検証可能
